# Jinguzhou 0.2 Developer Experience Release Plan

This plan defines the 0.2 release scope. The release moves Jinguzhou from a
first preview to a repository that another developer can install, run, and test
without reading the source first.

## Release Goal

Reduce the setup work required to install Jinguzhou, generate a config, run the
gateway, and validate policy behavior.

The 0.2 release covers four setup tasks:

- How do I install it?
- How do I initialize a working config?
- How do I run the gateway locally?
- How do I validate that policy, audit, approval, and tool enforcement work?

## Positioning

Jinguzhou 0.2 is a developer preview of a policy gateway for LLM applications,
agents, and tool execution.

Primary users:

- LLM application developers
- Agent framework developers
- Platform and infrastructure engineers
- Security engineers evaluating LLM governance controls

## Today Release Scope

### P0: Must Ship

- Version bump from `0.1.0` to `0.2.0`.
- `CHANGELOG.md` entry for 0.2.
- Updated `README.md` quick start for:
  - editable Python install
  - CLI help
  - local validation
  - gateway startup
- Developer setup guide in `docs/`.
- Config validation command:
  - load runtime config
  - load referenced policy files
  - build the gateway runtime without starting the server
  - print a machine-readable success result
- Project initialization command:
  - create a starter `jinguzhou.yaml`
  - optionally create starter `rules/` directory
  - avoid overwriting existing files unless `--force` is provided
- Docker quick start:
  - `Dockerfile`
  - `.dockerignore`
  - documented `docker build` and `docker run` commands
- Release validation script updated for `0.2.0`.
- Tests for new CLI commands.

### P1: Ship If Time Allows

- `docker-compose.yml` with gateway service. Done after P0.
- `examples/dev_quickstart/` with a minimal working config and README. Done after P0.
- CLI output mode option for JSON vs human-readable text.
- More helpful error messages for bad config or missing policy files.

### P2: Defer After 0.2

- npm package published to the public npm registry. Package scaffold added; registry publishing remains an account operation.
- PyPI publication automation. GitHub trusted-publishing workflow added.
- Helm chart. Starter chart added.
- Web dashboard. Local status page added at `/dashboard`.
- Postgres audit backend. Optional backend added.
- Slack or Teams approval inbox. Extension endpoint added at `/approvals/pending`; provider-specific adapters remain backlog work.
- Multi-tenant auth, RBAC, OIDC, or SSO. Admin API key guard added for local control-plane endpoints; full identity integration remains backlog work.

## npm / Node Direction

The 0.2 release leaves room for npm distribution. Publishing a full npm package
is separate from this release unless the Node packaging is already in place.

Recommended approach:

- Reserve a package name such as `@jinguzhou/cli`.
- Use the npm package as a thin developer launcher around the Python package,
  Docker image, or remote gateway.
- Target commands:
  - `npx @jinguzhou/cli init`
  - `npx @jinguzhou/cli gateway`
  - `npx @jinguzhou/cli validate`

The first npm package should not duplicate the policy engine in TypeScript. Its
job is to reduce setup work while keeping enforcement behavior in one
implementation.

## Proposed CLI Surface

Existing:

```bash
jinguzhou check-input
jinguzhou check-output
jinguzhou check-tool
jinguzhou gateway
jinguzhou audit query
jinguzhou audit replay
jinguzhou approval issue
jinguzhou approval decode
jinguzhou version
```

Add for 0.2:

```bash
jinguzhou init --output jinguzhou.yaml
jinguzhou validate-config --config jinguzhou.yaml
```

Optional if time allows:

```bash
jinguzhou doctor
```

## Developer Installation Targets

Python local development:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
jinguzhou version
```

Planned PyPI installation:

```bash
pip install jinguzhou
jinguzhou init
```

Docker:

```bash
docker build -t jinguzhou:0.2.0 .
docker run --rm -p 8787:8787 \
  -e OPENAI_API_KEY="$OPENAI_API_KEY" \
  -e JINGUZHOU_APPROVAL_SECRET=change_me \
  jinguzhou:0.2.0
```

Planned npm launcher:

```bash
npx @jinguzhou/cli init
npx @jinguzhou/cli gateway
```

## Acceptance Criteria

The 0.2 release is ready when:

- `python3 scripts/validate_release.py` passes.
- `jinguzhou version` prints `0.2.0`.
- `jinguzhou init --output /tmp/jinguzhou.yaml` creates a valid config.
- `jinguzhou validate-config --config /tmp/jinguzhou.yaml` succeeds.
- `docker build -t jinguzhou:0.2.0 .` succeeds.
- The README covers installation, validation, and local gateway startup.
- The release notes list the developer-facing changes.

## Release Checklist

1. Implement P0 items.
2. Run unit tests.
3. Run release validation.
4. Build Docker image locally.
5. Update `CHANGELOG.md`.
6. Commit as `Prepare 0.2.0 release`.
7. Tag `v0.2.0`.
8. Push `main` and the tag.
9. Create GitHub release notes from the changelog.

## Out Of Scope For Today

Do not block 0.2 on control-plane features. These belong to later releases:

- persistent approval queue UI
- hosted dashboard
- organization management
- SSO/RBAC
- SIEM integrations
- signed audit chains
- production Kubernetes manifests

## 0.3 Follow-Up Direction

After 0.2, prioritize integrations and storage:

- MCP adapter package and examples
- LangChain middleware package and examples
- Node/npm launcher
- Docker image publication
- Postgres audit backend
- lightweight dashboard for audit query and approval review
