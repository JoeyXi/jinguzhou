# Jinguzhou 0.3.0-alpha Agent Ecosystem Plan

This alpha release focuses on the shared execution layer that agent integrations
will use before framework-specific packages are added.

## Scope

- Adapter foundation
- JSONPath-like extractor completion
- First-pass file, network, and database policy packs

## Step 1: Adapter Foundation

Deliverables:

- public `jinguzhou.adapters` namespace
- `NormalizedToolCall` compatibility model
- direct `ToolAdapterRegistry.normalize_tool_call(...)`
- extraction for:
  - OpenAI tool calls
  - MCP content blocks
  - MCP JSON-RPC `tools/call`
  - LangChain tool calls
  - LlamaIndex-style top-level tool call payloads
  - OpenAI Agents-style `function_call` output items

Acceptance:

- all supported shapes normalize to the same internal tool-call object
- adapter configs continue to provide extractor mappings automatically
- existing gateway behavior remains compatible

## Step 2: JSONPath Extractor

Deliverables:

- dot notation
- bracket notation
- quoted keys
- array indexes
- negative array indexes
- `[*]` wildcard
- object wildcards
- recursive key lookup with `$..key`

Acceptance:

- nested file paths, URLs, SQL statements, and command strings can be extracted
  from framework-specific payloads without matcher changes
- invalid extractor paths fail clearly when resolved directly

## Step 3: Policy Packs

Deliverables:

- `rules/tool_file_access.yaml`
- `rules/tool_network_access.yaml`
- `rules/tool_database_access.yaml`

Acceptance:

- file policy blocks system writes and reviews secret/user-data access
- network policy blocks metadata and private network targets
- database policy blocks destructive operations and reviews mutations/exports

## Step 4: Release Validation

Deliverables:

- unit tests for adapter foundation and JSONPath behavior
- policy tests for file/network/database packs
- release validation check for v0.3 policy packs
- changelog and validation docs updated
